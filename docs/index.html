Core Concepts at a Glance

Content Addressable ID (CID)

Every file is identified by its SHA-256 hash. Same bytes → same CID → stored once. No filenames, no paths, just content.

Tenant Partitioning

Storage key format: tenant_{ID}/{CID}. TenantID resolved automatically from request context. Zero configuration for callers.

Automatic Deduplication

Upload same file twice → stored once per tenant. Write skipped on duplicate. Caller gets same CID back instantly.

Point-in-Time Restore

Daily restic snapshots to Backblaze B2. Restore any file from any day in the last 12 months. Encryption built-in.

Storage Flow Example

User uploads audio file "interview.mp3"

  1. Caller: Invokes StoreAsync(stream, "interview.mp3", "audio/mpeg")
  2. S3ObjectStorageProvider: Reads stream → computes SHA-256 hash
  3. CID Generated: "a1b2c3d4e5f6g7h8..." (SHA-256 of audio bytes)
  4. Storage Key: "tenant_42/a1b2c3d4e5f6g7h8..." (includes tenant context)
  5. Dedup Check: Does this key already exist in MinIO? (usually NO on first upload)
  6. Upload: S3ObjectService.UploadAsync(key, bytes)
  7. Return: "a1b2c3d4e5f6g7h8..." returned to caller
  8. Backup: File automatically included in next daily restic snapshot
On Duplicate Upload: If the same audio bytes are uploaded again by the same tenant, the dedup check returns immediately without uploading. The caller gets the same CID back instantly. This saves both storage and bandwidth.

API Reference

Storage V1 provides a simple, consistent interface through the IObjectStorageProvider contract. All callers interact through this interface regardless of backend.

IObjectStorageProvider Interface

StoreAsync Method

Task<string> StoreAsync(
    Stream stream,
    string? fileName,
    string? mimeType,
    StoreOptions? options = null,
    CancellationToken ct = default);

Description: Stores a file and returns its Content Addressable ID (CID).

  • stream: File content stream (read from beginning)
  • fileName: Optional filename for logging
  • mimeType: MIME type (e.g., "application/pdf")
  • options: Per-request overrides for ACL, storage class
  • ct: Cancellation token
  • Returns: CID (SHA-256 hash as lowercase hex string)
  • Throws: StorageCapacityException (disk full), InvalidOperationException (other errors)

Behavior: Reads stream, computes SHA-256, checks for duplicate. If new, uploads to storage backend. Always returns CID (immediately on duplicate).

Example:

var fileStream = System.IO.File.OpenRead("document.pdf");
var cid = await _storage.StoreAsync(
    fileStream,
    "document.pdf",
    "application/pdf");
// cid = "a1b2c3d4e5f6..." (SHA-256 hash)

GetStreamAsync Method

Task<Stream> GetStreamAsync(
    string cid,
    CancellationToken ct = default);

Description: Retrieves a file stream by CID for the current tenant.

  • cid: Content Addressable ID from StoreAsync
  • ct: Cancellation token
  • Returns: Open stream positioned at beginning
  • Throws: FileNotFoundException (CID not found), UnauthorizedAccessException (cross-tenant access)

Behavior: Resolves storage key using current tenant context + CID. Retrieves from backend. Streams directly (no buffering).

Example:

var stream = await _storage.GetStreamAsync("a1b2c3d4e5f6...");
var bytes = new byte[stream.Length];
await stream.ReadAsync(bytes);
// bytes now contain the file content

ExistsAsync Method

Task<bool> ExistsAsync(
    string cid,
    CancellationToken ct = default);

Description: Checks if a file exists for the current tenant.

  • cid: Content Addressable ID
  • ct: Cancellation token
  • Returns: true if exists, false otherwise

Behavior: Resolves storage key using current tenant context. Checks backend without transferring data.

Example:

if (await _storage.ExistsAsync("a1b2c3d4e5f6..."))
{
    var stream = await _storage.GetStreamAsync("a1b2c3d4e5f6...");
}

Exceptions

StorageCapacityException

Thrown when disk usage exceeds WriteStopThresholdPercent. Callers should return HTTP 507 (Service Unavailable) to clients.

try
{
    var cid = await _storage.StoreAsync(stream, "file.pdf");
}
catch (StorageCapacityException ex)
{
    return StatusCode(507, new { error = "Storage service temporarily unavailable" });
}

Disk Capacity Guard

S3ObjectStorageProvider automatically checks disk usage before every write operation. If usage exceeds WriteStopThresholdPercent (default 90%), a StorageCapacityException is thrown, preventing writes and returning HTTP 507 to clients.

Why 90%? Prevents catastrophic full-disk scenarios. At 95% usage, systems can become unstable. A 90% threshold gives ops 5% headroom to respond.

Best Practices

For Developers

✅ DO: Store Once, Reference Many

Upload file → get CID → save CID in database → reference CID for all future operations.

// Upload
var cid = await _storage.StoreAsync(audioStream, "audio.mp3");
await _db.SaveAudioReference(conversationId, cid);

// Later: retrieve
var savedCid = await _db.GetAudioReference(conversationId);
var stream = await _storage.GetStreamAsync(savedCid);

✅ DO: Handle StorageCapacityException

Disk full is not a fatal error. Return HTTP 507 and let clients retry.

try
{
    var cid = await _storage.StoreAsync(stream, fileName);
}
catch (StorageCapacityException ex)
{
    _logger.LogError(ex, "Storage full");
    return StatusCode(507, new { error = "Storage temporarily unavailable" });
}

✅ DO: Use Streaming APIs

Never load entire file into memory. Use Stream directly.

using var file = System.IO.File.OpenRead(largeFile);
var cid = await _storage.StoreAsync(file, "large.bin");
// Stream consumed without buffering entire file

❌ DON'T: Assume Filenames Are Unique

Same filename uploaded twice might have different content. Always use CID.

// Wrong: comparing filenames
if (filename == "report.pdf") { ... }

// Right: comparing CIDs
if (cid == previousCid) { /* dedup detected */ }

❌ DON'T: Log Raw File Content

Storage V1 already prevents this, but never log file bytes in your code either.

// Wrong
_logger.LogInformation("Storing file: {content}", fileBytes);

// Right
_logger.LogInformation("Storing file: {cid} {name} {size}", cid, fileName, fileBytes.Length);

For Operations

✅ DO: Monitor Disk Usage

Set Prometheus alerts at 70%, 85%, 95%. Respond to 85% alert immediately.

✅ DO: Test Backups Regularly

Monthly restore test from restic. Verify encryption password is accessible.

✅ DO: Size Disk for 2× Peak Need

If forecasting 2.2TB peak, provision 4.4TB to stay below 70% threshold.

❌ DON'T: Ignore 507 Errors

If clients start seeing 507 (Service Unavailable), disk is likely full. Don't wait for the alert. Act immediately.

❌ DON'T: Change Configuration During Load

Endpoint, access keys, etc. are read at startup. Redeployment required. Do this during low-traffic window.

For Security & Compliance

✅ DO: Rotate Credentials Regularly

S3 access keys should rotate every 90 days minimum.

✅ DO: Enable Encryption at Rest

MinIO supports encryption. Backblaze B2 encrypts by default. Use both.

✅ DO: Audit Access via Logs

Review logs for unusual access patterns. CID hash + tenant + timestamp sufficient for audit trail.

❌ DON'T: Store Sensitive Secrets in Code

All credentials come from environment variables. Never commit to repo.

❌ DON'T: Skip RESTIC_PASSWORD Backup

Frequently Asked Questions

General Questions

Q: What is a CID?

A: Content Addressable ID. It's the SHA-256 hash of the file bytes. Same file → same CID, always. Used for deduplication and content integrity verification.

Q: Can I access files from another tenant?

A: No. Storage key includes tenant ID: tenant_{ID}/{CID}. Cross-tenant access is cryptographically impossible. The system checks tenant context on every request.

Q: What happens if I upload the same file twice?

A: First upload: file stored. Second upload: storage backend returns immediately (dedup hit). Caller gets same CID back. Zero additional storage cost.

Q: Can I delete files?

A: Not directly via IObjectStorageProvider (current version). Files are immutable once stored. Lifecycle policies handle archival/expiration. Full delete support planned for V2.

Operational Questions

Q: What happens if the disk fills up?

A: At 90% (default threshold), StoreAsync throws StorageCapacityException. HTTP 507 returned to clients. Automatic safeguard prevents catastrophic disk-full scenarios.

Q: How do I monitor disk usage?

A: Query minio_node_disk_used_bytes / minio_node_disk_total_bytes * 100 from Prometheus. Set alerts at 70%, 85%, 95%. Health check endpoint: /minio/health/ready returns 503 when disk is critically full.

Q: How long are backups kept?

A: Daily snapshots: 7. Weekly: 4. Monthly: 12. Annual: 2. Total retention: 12+ months. Restic automatically prunes older snapshots per retention policy.

Q: Can I restore a single file?

A: Yes. restic snapshots are per-file retrievable. Contact ops team with CID + date range to restore.

Cost Questions

Q: Which backend is cheapest?

A: Backblaze B2: ~$0.006/GB/month for storage. AWS S3 ~$0.023/GB/month. MinIO (self-hosted): only disk cost. Choose based on infrastructure (on-prem vs cloud), team capacity (ops overhead), and scale.

Q: How much can I save with deduplication?

A: Depends on file overlap. Payroll templates: 30% savings. Audio: 10-20%. AI outputs: 30-50%. Conservative estimate: 20-30% across mixed workloads.

Security Questions

Q: Is my data encrypted?

A: In transit: HTTPS (S3 protocol). At rest: depends on backend. MinIO can enable encryption. Backblaze B2 encrypts by default. AWS S3 requires opt-in. Recommend: enable all layers.

Q: Are file bytes logged?

A: Never. Logs contain only: CID (hash), TenantID, FileName, MimeType, operation type, timestamp. No raw file data ever logged. GDPR/HIPAA compliant.

Q: What if my access key is compromised?

A: Rotate immediately. Old key: no longer works. New key: deployed via environment variables. Restart apps. Restic password should be independent secret.
